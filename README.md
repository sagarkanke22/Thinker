# LogicLive

> Server-Driven UI platform for internal tools. Business logic lives in a
> database — edited by hand, generated by AI via **`/editor`**, or planned with
> an AI discovery chat via **`/plan`**. No redeploy. Change is live in under a second.

Built for the **Damco Builder Challenge · Track A (Engineer)**.

---

## The Problem

### Problem 1 — Planning

Before writing any widget, a developer spends time answering questions they
shouldn't have to answer alone.

Which table has this data? Which column? What does "this month" mean exactly?
Table or chart? Which columns does the user need?

That time should be spent understanding the problem. Instead it gets spent
guessing. And if you guessed wrong, you find out after the code is written.

**`/plan` fixes this.** Two AI agents handle the schema questions and the UI
questions before a single line is written. The developer's job is only to
understand and confirm — not to guess and discover.

### Problem 2 — Execution

After four years of writing internal tools, I noticed that the code which
actually changed between requests was always the same 3 or 4 lines. A SQL
query. A filter. A column name.

But a developer's time wasn't spent on those 3 lines. It was spent on
everything around them — where to write the code, how to structure it, how to
test it, how to get it deployed. The actual problem-solving took 5 minutes.
The surrounding process took the rest of the day.

**`/editor` fixes this.** The business logic lives in a database, not in a
codebase. You open a browser, edit those 3 lines directly, save, and the
change is live. No pipeline. No deploy. The developer's time goes entirely to
solving the problem — nothing else.

## How It Works

Three pages. One flow.

| Page | Who uses it | What it does |
|---|---|---|
| **`/plan`** | Developer | Describe what to build. Two AI agents ask the right questions before any code is written. One click drafts the widget and opens the editor. |
| **`/editor`** | Developer | Monaco editor + live preview + AI chat in one screen. Ctrl+S saves to the DB. Preview reloads in under a second. No deploy. |
| **`/app`** | End user | The running tool. No code, no config — just the live result. |

The flow is always: **Plan → Write Code → It's Already Live.** There is no
deploy step. That's the point.

## The AI Layer

### `/plan` — Question-First Protocol

Two specialist agents activate when you describe a requirement:

- **@backend** — your data engineer. Has the full database schema before the
  conversation starts. Asks about tables, columns, filters, aggregation, time
  periods. Speaks in terms of data, not UI.

- **@frontend** — your UI developer. Knows every widget type and what each
  one can and cannot do. Asks about widget type, columns, interactivity,
  highlighting. Speaks in terms of UI, not data.

They are deliberately separated. They ask different questions. They don't
duplicate each other.

**Why this matters — the hallucination problem.** The most common failure mode
with AI code generation is hallucination: the AI looks confident, writes code
that looks right — but the SQL references a column that doesn't exist, or the
widget type doesn't support the layout it assumed. That happens because the AI
made a plan before it had enough information.

LogicLive solves this with a **question-first protocol**. Before either agent
writes a single line of code:

- The @frontend agent runs a **CAPABILITY GAP CHECK** against the widget
  registry — the definitive list of what each widget type can and cannot do.
  If your requirement needs something a widget doesn't support, the agent
  surfaces that gap before the plan is created.
- The @backend agent asks: "Do I have enough schema information to write
  correct SQL right now?" If not, it emits CONTEXT QUESTIONS — not a guess,
  not a draft. Questions. The Build button stays suppressed until every open
  question is answered.

Every fact in the final plan is cited — FROM WHICH TABLE, FROM WHICH COLUMN,
FROM WHICH WIDGET CAPABILITY. If the agent cannot cite it, it asks. It does
not guess.

**When you're satisfied**, click **Finalize Plan**. A two-stage synthesis runs:

1. @backend produces a DATA SPEC — source table, columns, SQL approach, filters, grouping.
2. @frontend reads that spec and produces a UI SPEC + a drafted `render()` function.
   Column names match. Date filters match. The code is correct because both
   agents built on each other's verified work.

Click **Build →** — widget created in the DB, code saved, editor opens.
Preview loads immediately. No second AI round.

### `/editor` — AI Chat Panel

Three modes, strictly scoped to the current widget:

- **Generate** — describe what you want in plain English. AI rewrites the
  entire `render()` function. Each response has its own Apply button — you
  choose what goes into the code.
- **Explain** — AI reads the code and walks through it line by line in plain
  English. Useful when someone else wrote the function or you're returning
  after a month.
- **Review** — AI reads the code looking for problems: missing error handling,
  slow SQL, edge cases. Returns specific findings with suggested fixes.

Before every AI call, the backend assembles full context: current code,
database schema, 2–3 sample rows, widget base_config, full conversation
history. The AI sees all of it every time. That's why it can write correct SQL.

## AI Safety

- `logic_code` is the **only field AI can write** — `base_config` (layout) is
  deliberately excluded from the `SaveRequest` model. This is the key design
  decision: AI handles logic, developers own layout.

  Tools like Bolt and v0 let AI rewrite the entire UI, so your layout shifts
  between sessions. Columns disappear. Widgets move. Users notice. Developers
  get blamed. LogicLive enforces the line in the API.

- Every generated function is validated for an explicit `return` statement
  before being saved; one auto-retry if missing.

- **Sandbox**: subprocess with `RLIMIT_AS` (128 MB), 5 s timeout, restricted
  builtins, allowlisted imports (`json`, `datetime`, `math`, `sqlite3`,
  `statistics`, + any stdlib via explicit `import`). Each render call gets a
  fresh, isolated subprocess. It cannot crash the server. If it crashes or
  times out, the error is shown as an inline card — the rest of the page keeps
  working.

## The Backend (Plain Terms)

When a widget needs to render:

1. Read the widget's Python function from the database.
2. Spawn a brand new subprocess just for this call.
3. Run the function inside it with a 5-second timeout and 128 MB memory limit.
4. Collect the JSON output and return it to the browser.

The AI provider is pluggable via one env variable — switch between Claude
(Anthropic API) and a fully local Ollama model. Same prompt pipeline either way.

## Installation Guide

### Step 1 — Check Prerequisites

Verify you have the required versions:

```bash
python3 --version      # need 3.11+
node --version         # need 18+
npm --version          # comes with Node
```

If Python 3.11 is missing: [python.org/downloads](https://www.python.org/downloads/)  
If Node 18 is missing: [nodejs.org](https://nodejs.org/) (LTS version)

---

### Step 2 — Clone the Repo

```bash
git clone https://github.com/sagarkanke22/Thinker.git
cd Thinker
```

---

### Step 3 — Set Up the Backend

```bash
# create isolated Python environment
python3 -m venv .venv

# install the logiclive package + all dependencies
.venv/bin/pip install -e ".[dev]"
```

---

### Step 4 — Configure AI Provider

```bash
cp .env.example .env
```

Open `.env` and choose one option:

**Option A — Anthropic API (Claude)** ← recommended for best results
```env
LLM_PROVIDER=anthropic
ANTHROPIC_API_KEY=sk-ant-...
ANTHROPIC_MODEL=claude-sonnet-4-6
```
Get a key at [console.anthropic.com](https://console.anthropic.com) → API Keys.

**Option B — Ollama (fully local, no API key, no data leaves your machine)**
```bash
# install Ollama: https://ollama.ai/download
ollama pull qwen2.5-coder:7b
ollama serve                          # keep this running in a separate terminal
```
```env
LLM_PROVIDER=ollama
OLLAMA_MODEL=qwen2.5-coder:7b
```

> The editor and end-user view work without any AI provider configured.
> Only the AI chat panel and `/plan` discovery chat require one.

---

### Step 5 — Seed the Database

```bash
.venv/bin/python examples/seed.py
```

This creates `logiclive.db` with a demo `sales_report` screen and sample
orders data. You only need to run this once (or any time you want a fresh DB).

---

### Step 6 — Start the Backend

Open **Terminal A**:

```bash
source .venv/bin/activate
uvicorn logiclive.app:app --reload --port 8000
```

Verify it's running:
```
http://localhost:8000/health   →  {"status": "ok"}
```

---

### Step 7 — Start the Frontend

Open **Terminal B**:

```bash
cd frontend
npm install        # first time only
npm run dev
```

---

### Step 8 — Open the App

| URL | What you'll see |
|---|---|
| `http://localhost:5173/plan` | Discovery chat — **start here** |
| `http://localhost:5173/editor?screen=sales_report` | Dev editor (3-pane) |
| `http://localhost:5173/app?screen=sales_report` | End-user view |
| `http://localhost:8000/health` | Backend health check |

---

### Troubleshooting

**`ModuleNotFoundError: No module named 'logiclive'`**  
Run `pip install -e .` from the project root with the venv active.

**`logiclive.db` not found**  
Run `python examples/seed.py` first.

**`ANTHROPIC_API_KEY` not set error**  
Make sure `.env` exists (copied from `.env.example`) and the key is set.

**Frontend shows blank page**  
Confirm the backend is running on port 8000. The Vite dev server proxies
`/api/*` to `localhost:8000` — if the backend is down, all API calls fail
silently.

**Ollama connection refused**  
Run `ollama serve` in a separate terminal before starting the backend.

## Honest Tradeoffs

**Sandbox** uses `RLIMIT_AS` + restricted builtins. Good for a single-team
internal tool. For hostile multi-tenant use, switch to containers or
Pyodide (WASM-Python).

**SSE fan-out** uses an in-process `asyncio.Queue` per subscriber.
Single-worker is fine. For `gunicorn -w N`, switch to Redis pub/sub or NATS.

**No logic version history.** Production would need a `widget_logic_history`
table — insert-on-save + rollback UI. Missing today.

**5 widget types** by design — not 50. Adding one costs ~50 lines:
`logiclive/widgets_registry.py` + `frontend/src/widgets/<Type>.jsx` +
one line in `frontend/src/widgets/index.js`.

## What's Next

**Multi-screen manager** — a list of all internal tools with create/rename/delete
and per-screen access control. Today screens are seeded directly to the DB.

**Drag-and-drop canvas layout** — 2D widget positioning. Today widgets stack
