# LogicLive

> Server-Driven UI platform for internal tools. Business logic lives in a
> database and can be edited by hand, generated by AI, or planned with an
> AI discovery chat — entirely on your machine, no API key required.

Built for the **Damco Builder Challenge · Track A (Engineer)**.

---

## What it is

Internal tools share a pattern: a handful of inputs, a table, a chart or two,
and some business logic that changes every sprint. LogicLive separates that
into two layers so only the fast-changing part needs to change:

| Layer | Who edits | How |
|---|---|---|
| **Widget library** (text / input / button / table / chart) | Framework devs | Code, PR, deploy once |
| **UI config** (`base_config`) | App devs | YAML / JSON, version-controlled |
| **Business logic** (`logic_code`) | App devs **or AI** | In-app editor or AI chat — no redeploy |

The frontend is a generic config-renderer. It knows five widget types and
nothing app-specific. All data fetching and business logic lives in one Python
function per widget, stored in SQLite, executed in a sandboxed subprocess per
render call.

**Backend** — FastAPI + SQLAlchemy + SQLite  
**Frontend** — React 18 + Vite 5 + Monaco + react-plotly.js  
**AI** — Ollama (local model, no API key, no data leaves your machine)

## Features

### End-user view (`/app`)
- Generic SDUI renderer — reads any screen from the DB, no app-specific code
- Input widgets drive reactive re-renders across dependent widgets (debounced SSE)
- Branded page header, cards layout, controls row separated from content

### Discovery chat (`/plan`)
- Two specialist AI agents: **@backend** (data expert) and **@frontend** (UI expert)
- First-turn interactive context form: agents ask clarifying questions with
  clickable option buttons and a "type your own" fallback
- Domain-separated: @backend asks about schema/filters/aggregation, @frontend
  asks about widget type/columns/interactivity — no duplication between agents
- **Finalize Plan** button triggers joint synthesis: backend emits a DATA SPEC,
  frontend reads it and emits a UI SPEC + a `BUILD:` hook with drafted code
- **Build →** button creates the widget, saves the drafted `logic_code`, and
  opens the editor — preview is live on the first click
- **Plan Pipeline** side panel: 5 connected step cards (Question → Data →
  UI → Code → Build) that fill in as the conversation progresses
- **SOLUTION PLAN** section emitted when a request is blocked — concrete DDL,
  scaffolding steps, or workarounds rather than dead ends
- Saved conversations persist across browser refreshes

### Editor (`/editor`)
- **Widget tree** — lists all widgets, supports drag-and-drop reordering,
  hover-to-delete
- **Monaco code editor** — Python with Ctrl+S save
- **Live preview** — embedded iframe that reloads on every save via SSE
- **AI chat panel** — multi-turn history per widget, three modes:
  `generate` (write/rewrite), `explain` (line-by-line walkthrough),
  `review` (bug + improvement notes). Per-turn Apply button. localStorage
  persistence keyed per `(screen_id, widget_id)`.
- Top nav with Plan / Editor links and "Preview app ↗" link

### AI safety
- `logic_code` is the only field AI can write — `base_config` (layout) is
  deliberately excluded from the `SaveRequest` model
- Every generated function is validated for an explicit `return` statement
  before being saved; one auto-retry if missing
- Sandbox: subprocess with `RLIMIT_AS` (128 MB), 5s timeout, restricted
  builtins, allowlisted imports (`json`, `datetime`, `math`, `sqlite3`,
  `statistics`, + any stdlib via explicit `import`)

## Prerequisites

1. **Python 3.11+**
2. **Node.js 18+ and npm**
3. **Ollama** — for AI features

   ```bash
   # Install: https://ollama.ai/download
   ollama pull qwen2.5-coder:7b
   ollama serve          # runs on http://localhost:11434
   ```

   The editor and end-user view work without Ollama. The AI chat and
   `/plan` discovery chat require it. Override the model via `OLLAMA_MODEL`
   in `.env`.

## Quickstart

```bash
# 1. clone
git clone https://github.com/sagarkanke22/Thinker.git
cd Thinker

# 2. backend
python3 -m venv .venv
.venv/bin/pip install -e ".[dev]"
.venv/bin/python examples/seed.py       # seed demo screen + orders data

# 3. terminal A — backend
.venv/bin/uvicorn logiclive.app:app --reload

# 4. terminal B — frontend
cd frontend && npm install && npm run dev
```

Open:
- **`http://localhost:5173`** — landing page
- **`http://localhost:5173/plan`** — discovery chat (start here)
- **`http://localhost:5173/app?screen=sales_report`** — end-user view
- **`http://localhost:5173/editor?screen=sales_report`** — dev editor

### Key endpoints

| Method | Path | Purpose |
|---|---|---|
| `GET`  | `/render` | Execute widget logic, return config-with-data |
| `POST` | `/action` | Run button widget logic (side-effect actions) |
| `POST` | `/save` | Update widget's `logic_code` |
| `GET`  | `/logic` | Read widget's current `logic_code` |
| `GET`  | `/screen` | List widgets on a screen (sorted by `sort_order`) |
| `GET`  | `/events` | SSE stream of `widget-changed` events |
| `POST` | `/widget` | Create a new widget |
| `DELETE` | `/widget/{screen}/{widget}` | Delete a widget |
| `PUT`  | `/screen/order` | Persist drag-and-drop reorder |
| `POST` | `/ai/generate` | AI generates `logic_code` — SSE stream |
| `POST` | `/ai/discuss` | Two-agent discovery chat — SSE stream |
| `GET`  | `/ai/conversations` | List saved plan conversations |

## Honest tradeoffs

**Sandbox** uses `RLIMIT_AS` + restricted builtins. Good for MVP; for
hostile multi-tenant use, switch to containers or Pyodide (WASM-Python).

**AI quality** depends on the local model. `qwen2.5-coder:7b` handles
small Python functions well. Swap providers in `logiclive/ai.py` — both
`call_ollama` and `stream_ollama` are thin async HTTP wrappers; any
OpenAI-compatible endpoint drops in with a few lines.

**SSE fan-out** uses an in-process `asyncio.Queue` per subscriber.
Single-worker fine; for `gunicorn -w N` switch to Redis pub/sub or NATS.

**5 widget types** by design — not 50. Adding one costs ~50 lines:
`logiclive/widgets_registry.py` + `frontend/src/widgets/<Type>.jsx` +
one line in `frontend/src/widgets/index.js`.

**No logic version history**. Production would need a
`widget_logic_history` table (insert-on-save + rollback UI).

**No layout editor** — `base_config` is YAML/JSON written by hand. This
is intentional: AI is deliberately blocked from writing layout config,
which sidesteps the Bolt/v0 failure mode where the LLM rewrites layouts
unpredictably between sessions.

## Project layout

```
logiclive/
  app.py              FastAPI app + all endpoints
  db.py               SQLAlchemy models (Widget, Conversation, Order)
  sandbox.py          Subprocess sandbox runner
  ai.py               Ollama client — generate + discuss (streaming)
  agents.py           @backend + @frontend specialist agent prompts
  widgets_registry.py KNOWN_TYPES + merge_render helper

  AppRenderer.jsx     Generic SDUI renderer (end-user view)
  App.jsx             Router + TopNav shell
  Editor.jsx          3-pane editor shell
  plan/PlanPage.jsx   Discovery chat + Plan Pipeline side panel
  widgets/            Text, Input, Button, Table, Chart
  editor/             WidgetTree, CodeEditor, LivePreview, ChatPanel
  api/client.js       fetch helpers

examples/
  seed.py             Seeds sales_report screen + orders table
tests/
  test_db.py  test_sandbox.py  test_render.py
```
