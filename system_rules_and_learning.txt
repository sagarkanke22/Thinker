## CODEBASE RULES — LogicLive (Damco Builder Challenge)

C014 | TYPE: ARCHITECTURE / VOCABULARY
       SDUI vocabulary is FIXED. Use these exact terms in code, comments,
       docs, and AI prompts:
         - widget_id     : unique within a screen
         - screen_id     : unique within an app
         - base_config   : declarative widget shape (YAML/JSON), DEV-WRITTEN
         - logic_code    : Python function string in DB, DEV-OR-AI written
         - render config : merged output of /render (base + data + style)
       WHY: Vocabulary drift breaks the contract between renderer, runtime,
            and AI prompts.

C015 | TYPE: ARCHITECTURE
       UI config (base_config) is HUMAN-ONLY. AI must NEVER write or modify
       base_config rows. "Deterministic shell, AI brain."
       WHY: Letting AI touch layout undoes the architecture decision that
            differentiates LogicLive from Bolt/v0.

C016 | TYPE: RUNTIME
       Every /render call = fresh sandboxed Python subprocess. Module-level
       state MUST NOT carry between handler invocations.
       WHY: Stale state across requests is the most expensive bug class.
       SEE ALSO: L032 (same pattern, inherited) — directly applicable.

C017 | TYPE: API CONTRACT
       /render signature is FIXED:
         GET /render?screen_id=X&widget_id=Y&params=<urlenc-json>
         Response: { type, props, data, style?, error? }
       WHY: Frontend is generic. Contract drift breaks every screen.

C018 | TYPE: AI-CODEGEN
       AI prompts for logic_code MUST include: DB schema for reachable
       tables, 1-3 sample rows per table, the widget's base_config, prior
       logic_code (if editing), and the /render response schema.
       WHY: Without grounded context AI invents column names → silent fail.

C019 | TYPE: AI-CODEGEN
       Generated logic_code MUST end with an explicit `return` of a dict
       matching the /render response shape. Runtime rejects code without
       explicit return.
       WHY: Implicit None returns produce confusing client errors.

C020 | TYPE: SAFETY
       All logic_code runs in a subprocess (MVP) with timeout + memory cap
       + import allowlist + no filesystem writes.
       WHY: Untrusted DB-stored code is the highest-risk surface.

C021 | TYPE: API GUARDS
       Per L026: add explicit endpoint-level guards on screen_id and
       widget_id inside the handler. Pydantic with default= silently
       passes missing fields — never trust validation alone.

L018 | DATE: 2026-04-06
      TYPE: GENERIC
      LEARNING: In Python, assigning to a module-level variable inside a nested block
                (e.g. elif inside a try inside main()) creates a LOCAL variable unless
                'global varname' is declared first. Without the global declaration,

L034 | DATE: 2026-05-13
      TYPE: GENERIC
      LEARNING: f-strings inside a dict/list literal are evaluated IMMEDIATELY at the
                line where the dict is constructed — NOT lazily when the dict is accessed.
                If any f-string references a variable defined AFTER the dict literal,
                a NameError results at runtime (not at parse time, so py_compile misses it).

      ROOT CAUSE: validations_map used f"...{_dd_plant_count+1}" but _dd_plant_count was

L035 | DATE: 2026-05-13
      TYPE: GENERIC / OOXML
      LEARNING: When extracting formula cells from OOXML using regex, NEVER use `.*?` with
                re.DOTALL between the cell opening tag and `<f>`. The dot-star spans across
                `</c>` boundaries and matches a self-closing empty cell's opening tag, then
                consumes through to a DIFFERENT cell's `<f>` element — producing a false
                formula-cell match with corrupted cell_attrs (includes trailing `/` from the
                self-closing tag).
                Fix: require `<f>` as the DIRECT first child:
                  r'<c r="([A-Z]+)2"([^/>]*)><f>([^<]+)</f>'
                Key patterns: `[^/>]*` (not `[^>]*`) excludes `/` from attrs capture;
                no `.*?` between `>` and `<f>`.
      ROOT CAUSE: `<c r="A2" s="2"/>` (empty cell) matched as formula cell because
                  `([^>]*)` captured ` s="2"/` and `.*?` jumped to N2's `<f>` tag.
                  Output: `<c r="A3" s="2"/>` (self-closing) + orphaned `<f>...</f></c>` = broken XML.

L036 | DATE: 2026-05-13
      TYPE: GENERIC / OOXML / ZIP
      LEARNING: When write_cells_to_excel is called once per sheet and each call's OUTPUT
                is passed as the next call's INPUT (chained), any patch applied to a SHARED

                This makes the patch idempotent regardless of call count.

L039 | DATE: 2026-05-31
      TYPE: CODEBASE / WIRE FORMAT
      LEARNING: When parsing a Server-Sent Events stream from fetch().body in the
                browser, the frame separator regex MUST accept both LF-LF (\n\n)
                and CRLF-CRLF (\r\n\r\n). Splitting on the literal string '\n\n'
                silently fails on CRLF streams because the two \n chars in
                '\r\n\r\n' are NOT consecutive (a \r sits between them) — the
                String.split('\n\n') returns the original buffer unchanged, all
                frames accumulate into one giant buffer, and only the trailing
                end-of-stream flush parses anything, with all data lines
                concatenated into a single mangled "frame".
                Symptom: chunks arrive, console shows correct SSE bytes, but
                event count stays at 0/1 and the UI never updates because
                onEvent never fires per token.
                Pattern: `buf.split(/\r?\n\r?\n/)` — handles both LF and CRLF
                Same pattern applies to the inner line splitter inside one
                frame: `frame.split(/\r?\n/)` not `frame.split('\n')`.
      ROOT CAUSE: Anthropic's HTTP SSE response uses CRLF line endings per RFC
                  2616 (HTTP/1.1), while sse-starlette in earlier versions used
                  LF. Browsers and proxies don't normalize SSE bodies. A parser
                  written against one server can silently break against another
                  even though the wire is RFC-compliant.
      HOW TO APPLY: Any custom SSE parser over fetch().body MUST split frames
                    on `/\r?\n\r?\n/` and lines on `/\r?\n/`. Never use the
                    literal-string '\n\n'. If parser must also be lenient on
                    `event:` headers (some servers omit the trailing space),
                    use `.startsWith('event:')` and `.slice(6).trim()`, not
                    `.startsWith('event: ')`.

L040 | DATE: 2026-05-31
      TYPE: PROMPT ENGINEERING
      LEARNING: When an LLM agent runs code in a sandbox with non-standard
                rules (pre-injected modules, blocked builtins, etc.), the
                rules MUST appear in the prompt with EXPLICIT WRONG/RIGHT
                pairs — bullet-point prose isn't enough. Claude (and most
                LLMs) default to Python muscle memory: `from datetime import
                datetime` then `datetime.strptime(...)`. Telling it "no
                imports allowed" eliminates the import line but it then
                writes `datetime.strptime(...)` against the bare module,
                which fails with "module 'datetime' has no attribute
                'strptime'". The fix that worked: side-by-side WRONG/RIGHT
                pairs for every common gotcha (datetime.datetime,
                datetime.timedelta, datetime.date, etc.) with the exact
                error message you'd see for each wrong form.
                Also: review/optimize/explain modes need explicit
                instructions to CONSULT the sandbox rules, not just have
                them in the prompt. A reviewer that doesn't know "look
                at the sandbox section" will give Python-default advice
                like "move imports to the top" which is wrong here.
      ROOT CAUSE: Generic "no imports" rule was in the prompt but the
                  positive form (how to USE datetime correctly without
                  importing) was only one example, not a full table. AI
                  followed the negative rule but pattern-matched on
                  standard Python usage for everything else.
      HOW TO APPLY: For ANY sandboxed exec environment:
                    1. List pre-injected names explicitly
                    2. For each, show WRONG (with exact error msg) and
                       RIGHT side by side
                    3. Add a "rule of thumb" sentence summarising the
                       pattern (e.g. "every datetime name starts with
                       datetime.")
                    4. In secondary modes (review, optimize, explain),
                       explicitly tell the AI to cross-check sandbox
                       rules before giving fix recommendations.
                    Silence on a constraint = AI defaulting to its
                    training distribution, which is general Python.
