LOCKED BUILD PROBLEM (2026-05-30)
═══════════════════════════════════════════════════════
WORKING NAME: LogicLive (placeholder — finalize before submission)

ONE-LINE PITCH:
A Server-Driven UI platform for internal tools where business logic
lives in a database (not source files) and can be edited by hand or
through a scoped AI chat. No deploy required for logic changes.

PROBLEM (the friction):
Every time an internal tool needs a logic change — "exclude refunds
from totals", "add a date filter" — it's a full dev cycle: branch,
edit code, PR, merge, deploy. For high-frequency-change tools at
data-heavy companies this kills velocity. Felt by user at MathCo
across multiple client engagements.

ARCHITECTURE (Server-Driven UI pattern):
- DB schema: widgets(widget_id, screen_id, base_config, logic_code)
- Frontend: generic config-renderer (knows 5–6 widget types, zero
  app-specific logic)
- API contract:
    GET /render?screen_id=X&widget_id=Y&params=...
  Backend:
    1. Fetch logic_code from DB
    2. Sandboxed exec(logic_code, params) → returns config-with-data
    3. Return as JSON
  Frontend:
    Renders the returned config using the generic component library
- Pattern is proven (Airbnb, Lyft, Spotify use SDUI on mobile);
  the novel combo here = SDUI + DB-resident logic + AI editor +
  internal-tool target audience

WHAT THE DEV BUILDS:
1. Defines UI screens by inserting rows into the widgets table
   (base_config is YAML/JSON declaring widget type + props + layout)
2. For each widget that needs logic, writes Python directly in an
   in-app code editor OR opens chat to ask AI to write it
3. Frontend hits backend on render → backend executes logic from DB
   → returns full config-with-data → frontend draws it

DIVISION OF LABOR:
- UI config         → Dev writes (deterministic, version-controlled)
- Business logic    → Dev or AI writes (both paths supported)
- Component library → Pre-shipped (5–6 generic widgets)
- Runtime exec      → Framework handles (sandboxed)

AI ROLE (scoped, not omniscient):
- Chat is bound to ONE widget's handler at a time
- Prompt context includes: DB schema, sample data, the widget's
  base_config, prior version of the handler
- Writes/edits ONLY the Python function for that handler
- Does NOT touch UI config (that's intentionally human-only)

DEMO STORY (for the ~5 min video):
- Open editor with empty app → paste a UI config (table + date input)
  → live preview renders empty widgets
- Open chat on the table widget → "fetch sales since 2026-05-01
  grouped by product" → AI writes function → preview updates with data
- User: "exclude refunds" → AI updates → preview refreshes — all
  without a deploy
- Final wow: add a chart widget by inserting one row + asking AI for
  the logic — live in <30s

WHY IT FITS DAMCO (4 criteria scored):
- Problem ID         : real friction lived at MathCo across projects
- Tech Understanding : SDUI architecture + runtime exec + schema-
                       aware AI codegen + sandboxing
- Scoping            : narrow widget library + scoped chat = ship-able
                       in 2–3 weeks
- Self-Awareness     : "Deterministic shell, AI brain" — clear,
                       defensible split. AI deliberately NOT touching
                       UI. Honest tradeoff list (below).

HONEST TRADEOFFS (call out in the video):
- Latency: every render = backend roundtrip → mitigate with caching
  for static widgets, debounce for inputs
- Sandbox: MVP uses subprocess; production needs containers / WASM
- Versioning: DB-edits affect users immediately; need audit log +
  rollback (history table of logic_code revisions)
- Debugging: stack traces point to DB rows, not files; need
  logic-id-aware logging layer
- Schema-aware AI: needs a context provider to inject DB schemas +
  sample data into prompts

SCOPE ESTIMATE (~16 person-days for MVP):
- Component library (text, input, button, table, chart)   : 3 days
- UI config schema + generic renderer                     : 2 days
- DB schema for widgets + logic                           : 1 day
- Runtime: fetch logic from DB + sandboxed subprocess exec: 2 days
- Chat panel scoped to a single handler                   : 2 days
- AI: prompt → valid Python (with schema context)         : 2 days
- Live preview / hot-reload on logic change               : 2 days
- Demo polish + video recording                           : 2 days

OUT OF SCOPE FOR MVP (flag as "next steps" in video):
- Real sandbox isolation (subprocess is fine for demo)
- Multi-user collaboration
- Auth / RBAC
- Production-grade audit log
- More than 5–6 widget types
- Visual layout editor (devs write YAML)

[Show README ASCII diagram on screen]
"The whole thing in one picture: browser hits Vite, proxied to

SCENE 4 — Tradeoffs (face cam, ~1.5 min)
───────────────────────────────────────────────────
"Three calls I'd defend on a whiteboard:
 1. Deterministic shell, AI brain. UI config is dev-only YAML. AI
    is structurally not allowed to touch it — the SaveRequest model
    has no base_config field. This sidesteps Bolt/v0's failure mode
    where AI rewrites layouts between sessions.
 2. Subprocess sandbox, not full isolation. Good for MVP and demos.
    For real multi-tenant hostile use, switch to containers or WASM.
 3. SSE for live preview, asyncio.Queue for fan-out. Works for a
    single uvicorn worker. Multi-worker needs Redis pub/sub — that's
    a 50-line swap, not a redesign."

───────────────────────────────────────────────────
SCENE 5 — Failure modes (face cam, ~1 min)
───────────────────────────────────────────────────
"Three failure paths the system handles gracefully:
 1. AI generates code that errors at runtime — sandbox catches the
    exception and the /render response carries {error: {type, message}}.
    Frontend shows a red widget; the rest of the screen still works.
 2. AI generates code with no `return` — server retries once with a
    fixup prompt. Test asserts this works.
 3. Ollama is down — /ai/generate streams an error event; the editor
    keeps the user's prior code intact and shows the error inline.
What's not handled: malicious user code escaping the subprocess. That's
the WASM / container story for v2."
