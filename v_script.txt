SECTION 1 — PROBLEM FRAMING  (0:00 – 2:30)
[Screen: README.md preview in VS Code — scroll to "Why We Built This"]
───────────────────────────────────────────────────────────────────

"Hi, I'm Sagar. I'm a fullstack developer.

For four years, I built a lot of applications from scratch —
ERP systems, sales dashboards, inventory reports, admin panels.

And I noticed a pattern.

Every single request looked different on the surface.
'Show me revenue by product.' 'Add a filter for last week.'
'Change the color on the top row.' Different every time.

But when I actually sat down to write the code — it was always
the same 3 or 4 lines. A SQL query. A column name. A filter.

That was the real work. 5 minutes of actual problem-solving.

The rest of the day? Gone. Figuring out where in the codebase
to put those 3 lines. Writing tests. Getting a review.
Waiting for CI. Deploying. Telling the user it's done.

--- WHY IS THIS HARD? ---

The hard part is not the code. The hard part is everything
around the code.

Internal tools change constantly. Every sprint, the business
wants a new filter, a new column, a new report. But each change
goes through the same full cycle — PR, review, CI, deploy.
That cycle was built for features that last years.
Not for a SQL filter that changes every week.

And there's a second problem. Before I can even write those
3 lines, the user gives me a vague requirement:
'Show me revenue for this month.'

And now I have to guess:
  - Which database table has the revenue data?
  - Which column exactly — is it gross or net?
  - What does 'this month' mean — last 30 days or calendar month?
  - Should I show a table or a bar chart?

I make assumptions. And if I guess wrong, I only find out
after I've already written the code, deployed it, and shown
it to the user.

These two problems — slow deployment and requirement guessing
— are what LogicLive solves.

[Switch screen: Show this simple flow on screen]

  BEFORE LogicLive:
  Requirement → Guess → Code → Test → PR → CI → Deploy → Wrong → Repeat

  WITH LogicLive:
  Requirement → AI asks questions → Confirmed plan → Code → Save → Live

The key insight: business logic for internal tools changes
constantly but follows a fixed pattern. So take it out of the
codebase entirely. Put it in a database. Let people edit it
directly in a browser. No pipeline. No deploy."

SECTION 2 — SYSTEM DESIGN  (2:30 – 8:30)
[Screen: README architecture diagram in VS Code]
───────────────────────────────────────────────────────────────────

"Let me walk you through how this actually works under the hood.

[Show architecture diagram on screen]

The system is simple at its core.
A React frontend talks to a FastAPI backend.
The backend reads from SQLite and runs Python code in a sandbox.
That's it.

The interesting part is what lives in that database.

Every widget — every table, chart, input on the screen —
has two things stored in SQLite.

The first is the layout config. What type of widget is this?
What columns should it show? What's the label? This is set once
by the developer and never changes at runtime.

The second is the logic — a single Python function.
This is the 3 lines I was talking about. The SQL query.
The filter. The calculation. This is what changes every sprint.
And because it lives in the database, you can edit it directly.

[Navigate to http://localhost:5173/app?screen=sales_report]

So when a user opens this page, what happens is:
the browser asks the backend for the list of widgets on this screen.
For each widget, it says — run the Python function and give me JSON.
The backend reads the function from SQLite, runs it in an isolated
subprocess, and returns the result.
The browser just renders whatever comes back.

The frontend has no idea what 'sales report' means.
It knows five widget types — text, input, button, table, chart.
Everything else is just data from the database.

[Navigate to http://localhost:5173/plan]

Now — before any of that code gets written, the developer
comes to this page. The /plan page.

Let me show you exactly how it works with a real example.

[Type in the chat: "I want to show revenue by product for this month"]

I just typed a requirement in plain English. Now watch what happens.

Two AI agents are reading this — one is focused on the database,
the other is focused on the UI. And instead of immediately
writing code, they come back with questions.

[Wait for questions to appear as clickable buttons]

See these clickable buttons? The data agent is asking me things like:
— When you say revenue, do you mean total sales or net after returns?
— What does 'this month' mean — the current calendar month, or last 30 days?

And the UI agent is asking:
— Do you want a table or a bar chart?
— Which columns should the user see?

I just click what fits. I don't need to type anything.

[Click through the options and submit]

Now I submit. The agents come back with a structured response.
They tell me exactly which table they'll use, which column,
what the SQL filter looks like, and what the widget will look like.
Every assumption is written out — nothing is hidden.

If something is wrong, I can correct it right here in the chat
before any code is written.

[Click Finalize Plan]

When I'm happy, I click Finalize. The agents now produce the
actual Python function — with the correct table name, the correct
column, and the correct date filter. Because they asked before
they assumed.

[Click Build →]

One click on Build and the widget is created in the database,
the code is saved, and the editor opens automatically.

[Navigate to http://localhost:5173/editor?screen=sales_report]

This is where you write and fix the code.

On the left is a list of all widgets on this screen.
Click any one and its code loads in the center.

The center is split — Monaco code editor on top, live preview
below. Make a change, press Ctrl+S, and the preview updates
in under a second. No deploy. No refresh. Just save and see.

On the right is an AI chat that knows exactly what widget
you're looking at — it has the current code, the database schema,
and sample data. Ask it to rewrite the function, explain it
line by line, or find bugs. Every response has an Apply button.
You choose what goes into the code.

And when a user changes a filter on the /app page —
say they pick a different date — the backend fires a small signal
to every widget on that screen saying 'go re-fetch your data.'
The whole page updates reactively. No page refresh. No websockets."

SECTION 3 — TRADEOFFS  (8:30 – 12:00)
[Screen: README "Honest Tradeoffs" section in VS Code]
───────────────────────────────────────────────────────────────────

"Let me be honest about the decisions I made and what I would
change.

--- DECISION 1: AI CANNOT WRITE LAYOUT ---

The most important design decision in this entire system:
AI can only write logic_code. It cannot touch base_config.

The SaveRequest API model has no base_config field.
This is enforced at the API level, not just by convention.

Why? Tools like Bolt and v0 let AI rewrite the entire UI.
Which means your layout shifts between sessions. Columns
disappear. Widgets move. Users notice. Developers get blamed.

LogicLive separates what changes constantly — the logic —
from what should stay stable — the layout. AI handles one.
Developers own the other.

What would change my mind? If base_config changes were versioned
and rollback was instant, I'd consider allowing AI to suggest
layout changes as drafts — never auto-applied.

--- DECISION 2: SUBPROCESS SANDBOX ---

Every render() call gets a fresh subprocess with RLIMIT_AS
(128 MB memory limit) and a 5-second timeout.

Why subprocess and not containers? Speed. A subprocess starts
in milliseconds. A Docker container takes seconds. For a tool
that re-renders on every input change, that latency matters.

The tradeoff: two widget subprocesses share the parent
server's filesystem. For a single internal team, this is fine.
For hostile multi-tenant use, you'd want containers or
Pyodide (Python running in WebAssembly — zero filesystem access).

What would change my mind? If this moved to a multi-tenant SaaS,
I'd switch to Pyodide immediately. Same Python API, but fully
sandboxed with no OS access at all.

--- DECISION 3: SSE OVER WEBSOCKETS ---

Server-sent events for reactivity — not websockets.

Why? SSE is unidirectional (server to browser), which is all
we need. Widgets never send data upstream — they only receive
re-render signals. SSE is simpler, works over HTTP/1.1, and
doesn't need connection management.

The tradeoff: SSE fan-out uses an in-process asyncio.Queue
per subscriber. Single-worker uvicorn — perfect. Scale to
multiple workers with gunicorn — fan-out breaks. You'd need
Redis pub/sub or NATS to broadcast across workers.

--- DECISION 4: SQLITE ---

SQLite for everything — widgets, conversations, orders.

Why? Zero setup. Single file. Works on any machine out of the
box. For a demo and single-server internal tool, it's the right
call.

What would change my mind? Team collaboration — multiple
developers editing widgets simultaneously. SQLite locks at the
file level. Switch to Postgres and you get row-level locking,
proper concurrent writes, and pgvector if you want semantic
search on widget code later.

--- WHAT'S MISSING ---

No logic version history. Production needs a
widget_logic_history table — insert-on-save, rollback UI.
If a developer saves a broken function, there's no undo today.

No authentication. Any user on the network can access /editor.
For internal use behind a VPN this is acceptable. For anything
else, add OAuth."

SECTION 4 — FAILURE MODES  (12:00 – 14:00)
[Screen: Live app — intentionally break something to demonstrate]
───────────────────────────────────────────────────────────────────

"Let me show you what happens when things go wrong.

--- FAILURE 1: BAD CODE IN THE WIDGET ---

[In /editor, write a function with a syntax error. Press Ctrl+S.]

The backend receives the save. Writes it to SQLite.
The SSE event fires. The preview re-renders.

But the function has a syntax error — so the subprocess crashes.
The error is caught, wrapped in JSON, and returned as an inline
error card inside that widget.

The rest of the page keeps working. Other widgets are unaffected.
Only this widget shows the error — with the exact Python
traceback so you know exactly what to fix.

[Fix the error. Press Ctrl+S. Preview shows the correct output.]

--- FAILURE 2: TIMEOUT ---

[Write a function with time.sleep(10). Press Ctrl+S.]

The subprocess starts. The function runs. After 5 seconds —
the process is killed. Timeout error card appears in the widget.
The server is not blocked. Other requests keep working.

--- FAILURE 3: AI GENERATES BAD CODE ---

If the AI generates a function without an explicit return
statement — which can happen — the backend catches this before
saving. It runs one automatic retry with a corrected prompt.
If the retry also fails, the error is returned to the chat.
The code is never saved to the DB.

--- FAILURE 4: /plan AGENT CANNOT ANSWER ---

If the @backend agent doesn't have enough schema information —
if a table doesn't exist, or a column name is ambiguous —
it does not guess. It emits a CONTEXT QUESTION and the Build
button stays disabled until the question is answered.

The system is designed to surface uncertainty early — before
code is written — not after.

--- WHAT BREAKS SILENTLY ---

SSE fan-out across multiple workers. If you run gunicorn with
more than one worker, each worker has its own in-process queue.
A save event goes to one worker. Browsers connected to other
workers don't receive it. The preview stops updating.
This fails silently — no error, just no update.
Fix: Redis pub/sub. One day of work."
