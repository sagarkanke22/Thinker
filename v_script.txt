INTRO  (0:00 – 1:00)

───────────────────────────────────────────────────────────────────

"Hi, I'm Sagar. This is LogicLive.

LogicLive solves two problems: Planning and Execution.

─── PROBLEM 1 — PLANNING ────────────────────────────────────────

Before writing any widget, a developer spends time answering
questions they shouldn't have to answer alone.

Which table has this data? Which column? What does 'this month'
mean exactly? Table or chart? Which columns does the user need?

That time should be spent understanding the problem — what the
user actually needs, what the data actually looks like, what
the right solution actually is.

Instead, it gets spent guessing. And if you guessed wrong,
you find out after the code is written.

/plan fixes this. Two AI agents handle the schema questions
and the UI questions before a single line is written. The
developer's job is only to understand and confirm — not to
guess and discover.

─── PROBLEM 2 — EXECUTION ───────────────────────────────────────

After four years of writing internal tools, I noticed that the
code which actually changed between requests was always the same
3 or 4 lines. A SQL query. A filter. A column name.

But a developer's time wasn't spent on those 3 lines.
It was spent on everything around them — where to write the code,
how to structure it, how to test it, how to get it deployed.

The actual problem-solving — the 3 lines — took 5 minutes.
The surrounding process took the rest of the day.

/editor fixes this. The business logic lives in a database,
not in a codebase. You open a browser, edit those 3 lines
directly, save, and the change is live. No pipeline. No deploy.
The developer's time goes entirely to solving the problem —
nothing else.

Let me show you both."

─── WHY THIS MATTERS: THE HALLUCINATION PROBLEM ──────────────────

"Let me take 30 seconds on something important.

The most common failure mode with AI code generation is
hallucination. The AI looks confident, writes code that looks
right — but the SQL references a column that doesn't exist,
or the param key is wrong, or the widget type doesn't support
the layout it assumed.

Why does that happen? Because the AI made a plan before it had
enough information. It filled in the gaps with plausible-sounding
assumptions and never flagged them as unverified.

LogicLive solves this with a question-first protocol.

Before either agent writes a single line of code, the system runs
a CAPABILITY GAP CHECK. The @frontend agent checks the widget
registry — the definitive list of what each widget type CAN and
CANNOT do. If your requirement needs something a widget doesn't
support, the agent surfaces that gap BEFORE the plan is created.

The @backend agent runs its own gate — 'Do I have enough schema
information to write correct SQL right now?' If the answer is no,
it emits CONTEXT QUESTIONS. Not a guess. Not a draft. Questions.
The Build button stays suppressed until every open question is
answered.

This is the key insight: the agents ask the right question so
that the plan is created correctly. Not 'generate something
plausible and let the developer find the bug.' Instead — 'refuse
to generate until we have verified answers.'

Every fact in the final plan is cited — FROM WHICH TABLE,
FROM WHICH COLUMN, FROM WHICH WIDGET TYPE CAPABILITY. If the
agent cannot cite it, it asks. It does not guess.

That is how we get a plan where the code is correct on the first
build — because the plan was not written until the plan could
actually be correct."

PART 5 — HOW IT ALL WORKS UNDERNEATH  (7:00 – 7:45)
[Talk through — no demo needed]
───────────────────────────────────────────────────────────────────

"Let me explain the backend in the simplest possible terms.

When a widget needs to render — when you open /app or when a
filter changes — the backend does four things:

  1. Read the widget's Python function from the database.
  2. Spawn a brand new subprocess just for this call.
  3. Run the function inside it with a 5-second timeout
     and 128MB memory limit.
  4. Collect the JSON output and return it to the browser.

That subprocess is isolated. It cannot crash the server. It
cannot access anything outside its allowed modules. It cannot
share state with another widget's subprocess. Every render is
completely fresh.

If the function crashes — syntax error, bad SQL, whatever —
the error is caught, wrapped in JSON, and returned as an inline
error card in the widget. The rest of the page keeps working.

If it takes longer than 5 seconds — timeout, killed, error shown.

The AI layer adds two things on top.
For the editor chat — ai.py builds a full context prompt and
streams Claude's response token by token to the browser.
For the plan chat — agents.py builds two separate prompts, one
per agent, with their respective domain knowledge, and streams
both responses in parallel.

The provider is pluggable — one env variable switches between
Claude via Anthropic API and a fully local Ollama model.
Same prompt pipeline either way."

─ DRAG-AND-DROP CANVAS LAYOUT

Today widgets stack vertically. Up and down.

The next big UI upgrade is a 2D canvas. You drag widgets not just

Put two charts side by side. Create a dense dashboard layout.

Under the hood, base_config gains x, y, width, height fields.

And the AI constraint stays. AI can only write logic_code.
It cannot touch layout config. The canvas is always exactly
what the developer designed.

─ TASK TRACKING TREE IN /plan

Right now /plan is a chat. Great for discovery. But when you're
building a screen with six widgets you lose track of where you are.

The next evolution adds a task tree alongside the chat:

  Sales Report
    ├── LIVE      input.start_date
    ├── LIVE      input.end_date
    ├── LIVE      text.total_kpi
    ├── BUILT     table.results
    └── PLANNING  chart.revenue  ← currently open

Each node has a status. When you click Build it flips to Built.
When the developer saves working code it flips to Live.

Zoom out one level and you get the full application tree:

  App
    ├── Sales Report        (4/5 widgets live)
    ├── Inventory Dashboard (2/3 widgets live)

───────────────────────────────────────────────────────────────────
PART 7 — TRADEOFFS  (9:00 – 9:30)
───────────────────────────────────────────────────────────────────

"Three honest tradeoffs.

The sandbox is process-level isolation. Two widget subprocesses
share the parent's filesystem. For a single-team internal tool
this is fine. For hostile multi-tenant use you'd want containers.
Known. Documented. Solvable.

SSE fan-out is in-process. One server worker — perfect. Scale
horizontally — add Redis pub/sub. One day's work.

No logic version history. Production needs a history table so
you can roll back a bad widget function. Missing today.

And the design decision I'll defend the most:

AI cannot write layout config. The save endpoint has no
base_config field. This is intentional.

Tools like Bolt and v0 let the AI rewrite the entire UI.
Which means your layout shifts between sessions. Columns
disappear. Widgets move. Users notice. Developers get blamed.

LogicLive separates what changes constantly — the logic —
from what should stay stable — the layout. AI handles one.
Developers own the other. That line is enforced in the API."
